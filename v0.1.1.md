# Mixi v0.1.1 - Roadmap \& Execution Plan

**Goal**: Transform MixiDAW v0.1.0 into a rigorously tested, professionally documented, and responsive platform.

## 1. Mobile Responsiveness (Forced View)

**Objective**: Make the Mixi interface usable (or at least explorable) on mobile devices without shattering the delicate WebAudio/Canvas integrations and absolute layouts designed for desktop.

* **Proposed Approach**:
  * Implement a top-level CSS transform/scale wrapper bounded by a `max-width` media query to shrink the desktop view gracefully.
  * Adjust touch/drag events in `useDrag.ts` and `JogWheel.tsx` to handle passive mobile touch events correctly.
  * If the styling alterations degrade desktop performance or break the Canvas calculations, we will fall back to a "Desktop Only" blockscreen warning for mobile users.

## 2. Homepage & UX Refinement

**Objective**: Elevate the first impression. The homepage and app UI need a comprehensive audit for spacing, animations, performance, and typography.

* **Proposed Approach**:
  * **Aesthetics**: Revisit `index.md` hero dimensions, contrast ratios, and improve glassmorphic properties.
  * **UX Flow**: Ensure hover states and micro-interactions perform cleanly without layout jank (CLS).
  * **Code Quality**: Audit CSS custom properties (`index.css`), removing redundant Tailwind classes.

## 3. Documentation Architecture & Deep Links

**Objective**: Structure VitePress routing so the Homepage (`/`), Docs (`/guide/`), and Web App (`/app/`) exist harmoniously on GitHub Pages.

* **Proposed Approach**:
  * Create a `docs/guide/` directory and populate it with technical manuals (Architecture, Skins, MIDI).
  * Update `.vitepress/config.ts` to add a `nav` bar linking strictly to `/guide/index.html` (the standalone Docs hub).
  * Verify GitHub Pages handles `/app/` alongside VitePress artifacts cleanly.

## 4. Comprehensive Testing Suite (The Core of v0.1.1)

**Objective**: Establish a professional TDD/BDD bedrock for audio accuracy, state management, and interface performance.

* **Proposed Approach**:
  1. **Unit Testing** (Vitest): Focus on the core Audio Logic (`BpmDetector`, `KeyDetector`) and Data Stores (`mixiStore`, `settingsStore`).
  2. **AI / Intents Engine**: Test `Blackboard` state snapshots and verify `Ghost Mutations` bounds statically.
  3. **E2E & UI Testing** (Playwright): Validate End-to-end boot sequences, Deck loading, and UI responsiveness.
  4. **Performance Benchmarking**: Automate checks for UI frame drop rates and AudioContext jitter.
